# Deviation Log — v0.3.0

## DEV-001: FossicStore.append return type

**Spec:** Design doc §3.1 shows `append() -> EventId`; §3.2 EventEmitter uses `_last_event_id: Optional[bytes]` and returns `bytes` from `emit_cycle_event`.

**Deviation:** `FossicStore.append` returns `bytes` (not `EventId`). Internally converts `causation_id: Optional[bytes]` to `EventId` via `EventId.from_hex()` for the Append constructor, then returns `eid.as_bytes()`.

**Rationale:** Design doc is internally inconsistent on this boundary (§3.1 says EventId, §3.2 uses bytes). Resolving in favor of bytes at all Cerebra-layer boundaries avoids requiring callers to import fossic types. All EventEmitter internals work correctly with bytes.

## DEV-002: FossicStore.take_snapshot return type

**Spec:** Design doc §3.1 shows `take_snapshot() -> SnapshotInfo`.

**Deviation:** Returns `Optional[SnapshotInfo]`, catching `NoEventsToSnapshotError` and returning `None`.

**Rationale:** The trigger function in §3.2 calls `store.take_snapshot(stream_id)` without error handling. A stream at zero events would raise if we propagated the exception. Returning None is the documented "graceful" behavior from the step prompt risk list: "handle NoEventsToSnapshotError gracefully (return early)."

## DEV-003: PHASE_6_EVENT_TYPES count is 22, not 23

**Spec:** Prior planning session referenced "23 event types" in the vocabulary doc.

**Deviation:** The vocabulary doc (`cerebra_phase6_event_vocabulary.md`) contains exactly 22 event types (22 `####` section headers). Both `inspector/event.py` and `cognition/_constants.py` define 22 types to match.

**Rationale:** Source-of-truth count from the vocabulary doc. Tests assert `len == 22`.

## DEV-004: fossic StoredEvent.payload is a method call

**Spec:** No prior specification; assumption was attribute access.

**Discovery:** `stored_event.payload` returns a callable; the payload dict is accessed via `stored_event.payload()`. Same for `stored_event.indexed_tags()`.

**Impact:** All tests accessing payload content use method-call syntax. Implementation is unaffected (reducer's `apply(state, payload)` receives the dict directly from fossic internals).

## DEV-005: CCE deduplication in fossic

**Spec:** Not specified in design doc.

**Discovery:** fossic applies Content-Addressed Event (CCE) deduplication: events with identical `(stream_id, event_type, payload, causation_id, branch)` are collapsed to a single stored event with the same EventId. Appending N identical events to a stream results in 1 stored event.

**Impact:** All test loops that append multiple events must use unique payloads (e.g., `{"_seq": i}` counter). Implementation is unaffected — cycle events naturally have unique payloads; lattice events coming from real cognition loops will differ. The FossicStore wrapper does not add any dedup-bypassing logic; unique payloads are the caller's responsibility.

## DEV-006: LLMAdapter ABC shape divergence (Step 2)

**Spec:** Step 2 prompt shows `complete()` and `complete_structured()` as abstract methods on `LLMAdapter`.

**Deviation:** Both methods are implemented as concrete methods on the ABC that raise `NotImplementedError`, not as `@abstractmethod`. `OllamaDirectAdapter` and `ProxyLLMAdapter` override them with real implementations.

**Rationale:** Making them `@abstractmethod` immediately broke all existing `MockAdapter` subclasses in `test_sku_classifier.py` (TypeError: can't instantiate abstract class without implementing abstract methods). Concrete-with-raise preserves the interface contract without requiring existing subclasses to implement signal-evaluation methods they don't need. The behavioral contract is identical; only the enforcement mechanism differs.

## DEV-007: Migration013 number consumed by predictions/outcomes (Step 3)

**Spec:** `v01_phase6_design.md §6.4` (Phase 8) reserved Migration013 for sessions + continuation_bundles tables.

**Deviation:** Phase 6 Step 3 consumes Migration013 for predictions + outcomes tables. Phase 8 will need Migration014+ for sessions/continuation_bundles.

**Rationale:** The design doc originally planned Migration012 to cover evaluations + predictions + outcomes together. Step 2 split evaluations into their own Migration012. Step 3's predictions/outcomes naturally became Migration013 following that split. The numbering shift is minor — Phase 8's migration author needs to be aware that 013 is taken.

## DEV-008: test_migration012.py test_predictions_table_not_created updated (Step 3)

**Spec:** `tests/unit/test_migration012.py` contained a guard test asserting that predictions/outcomes tables are NOT present after running migrations. This was valid when only Migration012 existed.

**Deviation:** After Migration013 landed, `run_migrations()` runs both 012 and 013, so predictions/outcomes DO exist after the call. The guard test was replaced with `test_migration012_description_references_evaluations` which verifies that Migration012's description correctly references evaluations (not predictions).

**Rationale:** The original test's intent (Migration012 does NOT create predictions/outcomes) is still correct — Migration012's SQL does not touch those tables. The test simply can't be expressed as "tables absent after run_migrations()" anymore. The step 3 test suite (test_migration013.py) now covers the predictions/outcomes creation guarantee.
