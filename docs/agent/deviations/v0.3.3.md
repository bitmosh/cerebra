## DEV-011: Prototype gate decision — Option A (no prototype)

**Gate:** Phase 8 §6.2 prototype gate decision, required at Phase 8 kickoff.

**Decision:** Option A — proceed directly to Phase 8 Step 1 implementation. No thin prototype inserted.

**Reasoning:**
- Phase 8 Step 1 is pure persistence foundation (RuntimeSession + SessionManager + Migration014). Zero integration risk — no LLM, no cycle composition, no event ordering concerns. There is nothing to prototype at this step.
- All Phase 6–7 component interfaces are established and individually tested: EventEmitter, FossicStore, PredictionPipeline, LeewayPreActionGate, EvaluationComposer. Their individual contracts are solid.
- Integration risks (LLM adapter under cycle-runtime load, event emission ordering in cycle loop, ContextPacket + working memory interaction) are concentrated in Step 2, not Step 1. Those are the appropriate point to validate composition.
- The methodology's two-attempt cap and cascade-halt rules provide in-process risk management for Step 2. If Step 2 surfaces blocking integration issues, Option B prototype work can be inserted between Step 1 and Step 2 without retroactively invalidating Step 1.
- The vocabulary spec resolved the key architectural ambiguity (stream pattern for SessionOpened) before any code was written: `session_id: str — UUID, also the stream's cycle_id segment`. This means `EventEmitter(store, session_id=session_id, cycle_id=session_id)` is canonical, not a placeholder.

## DEV-012: SessionOpened stream pattern — session_id IS the cycle_id segment

**Spec:** The Phase 8 kickoff prompt suggested a placeholder `session-<session_id>` as the cycle_id for session-level events, pending Step 2 reconciliation.

**Deviation:** The placeholder is not needed. The vocabulary spec (`cerebra_phase6_event_vocabulary.md`, SessionOpened entry) explicitly states: `session_id: str — UUID, also the stream's cycle_id segment`. The SessionOpened event lives on `cerebra/agent-trace/<session_id>` where the stream segment is the session_id itself, not a compound `session-<session_id>` string.

**Rationale:** EventEmitter is constructed with `cycle_id=session_id` for session-level events. This is not awkward — it's the canonical pattern per the vocabulary spec. CycleStarted events (Step 2) will live on `cerebra/agent-trace/<cycle_id>` where `cycle_id` is a separate generated identifier. Sessions and cycles have separate stream segments; they're not on the same stream.

## DEV-013: runtime_sessions table name (not sessions)

**Spec:** Phase 8 design doc §6.4 specified a `sessions` table for RuntimeSession persistence.

**Deviation:** Table named `runtime_sessions` instead of `sessions`.

**Rationale:** Migration009 (Phase 5) already created a `sessions` table for working-memory session tracking (`session_id, vault_path, status, started_at, last_active_at`). Adding a second `sessions` table with a different schema would cause an OperationalError at migration time. `runtime_sessions` avoids the collision while clearly expressing the domain.

**Impact:** All persistence helpers in `cerebra/cognition/session.py` reference `runtime_sessions`. No consumer code outside this module queries the table directly.
