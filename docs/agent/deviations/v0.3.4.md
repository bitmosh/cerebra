## DEV-014: CycleStarted causation_id = None (cross-stream deferred)

**Spec:** Vocabulary spec implies CycleStarted should chain causally from SessionOpened so the
full agent-trace causation tree is traversable.

**Deviation:** `CycleStarted` is emitted with `causation_id=None` in Step 2. This makes it the
root event of the cycle's fossic stream rather than causally linked to the session's stream.

**Rationale:** SessionOpened lives on `cerebra/agent-trace/<session_id>` and CycleStarted lives
on `cerebra/agent-trace/<cycle_id>`. These are different streams. Linking them requires the
CycleRuntime to receive the SessionOpened event_id from SessionManager, but Step 2 didn't modify
`open_session()` to return that id (that's Step 3's D3 fix). Making `causation_id=None` in the
meantime is a v0.1 deviation — the cycle stream is self-consistent, just not chained to the
session stream.

**Resolution:** Step 3 fixes this (DEV-018). `SessionManager.open_session()` returns
`(RuntimeSession, opened_event_id)` tuple. `CycleRuntime` uses `opened_event_id` as
`causation_id` for `CycleStarted`.

## DEV-015: SessionFlushed emitted by CycleRuntime, not SessionManager

**Spec:** Phase 8 Step 1 design implied `flush_session()` would emit the `SessionFlushed` event.

**Deviation:** `CycleRuntime.run()` emits `SessionFlushed` as the final event in the cycle
stream (causation=`CycleCompleted` event_id). `SessionManager.flush_session()` is DB-only (no
fossic emission).

**Rationale:** The causation chain requires `CycleCompleted → SessionFlushed`. `flush_session()`
doesn't have access to the `CycleCompleted` event_id, and adding that parameter would couple the
session lifecycle API to cycle internals. Emitting from `CycleRuntime` is cleaner — it owns the
cycle stream and has the causation id in scope.

**Impact:** `flush_session()` docstring updated to reflect DB-only semantics.

## DEV-016: render_template uses simple regex (no Jinja2 dep)

**Spec:** `simple.planning.v0.yaml` uses `{{ var }}` and `{{ prior_steps[N] }}` syntax,
implying Jinja2.

**Deviation:** `render_template()` in `cycle_config.py` is a custom regex implementation:
`re.sub(r"\{\{\s*(.*?)\s*\}\}", _replace, template)` with list-index support.

**Rationale:** Jinja2 is not in Cerebra's dependencies. Adding it requires dev approval per the
supply-chain safeguard protocol (active 2025–2026 supply-chain attack vector). The simple
renderer handles all patterns in v0.1 cycle configs: `{{ var }}` and `{{ var[N] }}` indexing.

**Limitation:** No Jinja2 filters, conditionals, or loops. v0.1 cycle configs are constrained to
simple variable substitution and list indexing. Phase 9 cycle configs that need richer templates
may require either Jinja2 dep approval or extended custom renderer.

## DEV-017: Episode DB write stubbed (D1 blocked on memory_records FK strategy)

**Spec:** Phase 8 Step 2 should write episode records after each accepted step. `MemoryWriteFromCycle`
event should reference a real `record_id` from a persisted memory record.

**Deviation:** Episode write is stubbed in Step 2. `MemoryWriteFromCycle` emits a synthetic
`record_id = _generate_id("episode")` with no corresponding DB row. The event fires, but
nothing persists to memory_records.

**Rationale:** `memory_records` table (Migration002) has NOT NULL FKs to `sources`, `documents`,
and `chunks`. Cycle episodes have none of those — they're produced by LLM calls, not ingestion
pipeline. Inserting into memory_records would require fake source/document/chunk rows (dirty)
or schema surgery (risky). A clean solution (Option B: separate `cycle_episode_records` table)
was posted to #brainstorm. Step 3 implements the chosen approach.

**Resolution:** Step 3 (DEV-019) implements episode persistence once the FK strategy is decided.

## DEV-018: StepExecutionFailed added to Phase 6 event vocabulary

**Spec:** Original Phase 6 vocabulary spec did not include `StepExecutionFailed`.

**Deviation:** `StepExecutionFailed` added to `PHASE_6_EVENT_TYPES` in `_constants.py` and
`inspector/event.py`. Payload: `{session_id, cycle_id, step_id, error_type, error_message,
retry_count, failed_at}`.

**Rationale:** LLM errors during cycle execution need a typed event for observability and
downstream processing (e.g., alerting, retry-policy evaluation in Phase 9+). The event was
specified in D5 of `phase_8_integration_decisions.md` as required, just not included in the
original vocabulary set.
