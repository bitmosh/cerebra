# Deviations ŌĆö v0.3.5a (Phase 8 D1 closure)

## DEV-020 ŌĆö cycle_episode_records rather than memory_records

**What changed:** Episodes from `CycleRuntime` are persisted to a separate `cycle_episode_records`
table (Migration016) instead of `memory_records`.

**Why:** `memory_records` carries NOT NULL FKs to `sources`, `documents`, and `chunks`. Cycle-generated
LLM output has no provenance in the ingestion pipeline and cannot satisfy those constraints without
fabricating foreign key rows. A separate table is the honest choice and avoids schema gymnastics.

**Impact:** Episodes are not yet visible to retrieval queries that target `memory_records`. Phase 10
consolidation will bridge selected episodes into `memory_records` once the provenance gap is resolved.
Until then, episode content is queryable via `EpisodeWriter.list_for_runtime_session()` and the
`cycle_episode_records` table directly.

**Scope:** Decision accepted in #brainstorm (phase_8_integration_decisions.md). No spec change required.

## DEV-021 ŌĆö EpisodeWriter placed in cerebra/cognition/ not cerebra/storage/

**What changed:** `EpisodeWriter` lives in `cerebra/cognition/episode_writer.py`, not the storage layer.

**Why:** Episodes are cycle-derived cognitive artefacts, not ingestion artefacts. Placing the writer
near `CycleRuntime` (its primary consumer) is more coherent than placing it with the ingestion
storage layer. The storage layer holds tables that back the ingestion pipeline; episodes are a product
of the cognition layer.

**Impact:** None ŌĆö a naming and placement convention only.

---

## DEV-022 ŌĆö ELEVATED_SALIENCE shares value with SYNTHETIC_ITEM_DEFAULT_SALIENCE (0.8)

**What changed:** `ELEVATED_SALIENCE = 0.8` was added to `_constants.py`. Its numeric value coincides
with `SYNTHETIC_ITEM_DEFAULT_SALIENCE = 0.8`.

**Why:** The two constants serve different semantic purposes: `SYNTHETIC_ITEM_DEFAULT_SALIENCE` is
the baseline for items inserted without an explicit score; `ELEVATED_SALIENCE` is the score assigned
when a cited `memory_record` is promoted into working memory after an accepted cycle step. They happen
to share a value at v0.3.5a but are expected to diverge as tuning proceeds.

**Impact:** None ŌĆö two separate named constants, same value.

## DEV-023 ŌĆö Citation extraction is best-effort regex; no LLM round-trip

**What changed:** `_extract_citations()` uses `re.findall(r"\brec_[0-9a-f]{12}\b", output_text)`.
It does not parse citations from structured LLM output or prompt the model to produce a citation list.

**Why:** LLM output in v0.1 does not reliably produce `rec_` IDs (the model has no context window
content that would give it those IDs). An empty citation list is the normal case. The regex is a
forward-compatible hook: when retrieval context is injected into step prompts (Phase 10), the
model will naturally produce `rec_` references and the regex will pick them up.

**Impact:** `cited_record_ids` will be `[]` for all cycle runs until Phase 10 wires retrieval
context into the prompt. No data loss.

---

## DEV-024 ŌĆö _boost_salience_for_cited silently skips missing memory_records

**What changed:** When `_boost_salience_for_cited()` encounters a cited `rec_` ID that doesn't
exist in `memory_records`, it continues without error rather than raising.

**Why:** Citations are extracted from LLM output via best-effort regex. A matched pattern may not
correspond to a real `memory_record` (hallucinated ID, format collision). Silent skip is correct
for a best-effort operation; the episode itself is still persisted.

**Impact:** None ŌĆö salience boost is an optimistic enhancement, not a required invariant.
