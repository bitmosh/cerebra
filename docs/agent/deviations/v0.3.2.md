## DEV-009: ConstitutionalRule.forbids() always returns False in v0.1

**Spec:** `v01_phase6_design.md §5.2` shows `c_rule.forbids(proposed_action)` as a gate step where constitutional rules can pre-emptively block actions.

**Deviation:** `ConstitutionalRule.forbids()` always returns `False` in v0.1. CONST-001 through CONST-005 are post-action output analyzers (`revokes_leeway_when` triggers check what output contains, not which capability is proposed). No pre-action capability-blocking rule shape exists in the current model.

**Rationale:** Forcing output-analysis rules to double as pre-action blockers would stretch `is_inviolable` and `applies_to` beyond their design intent. The constitutional rule model needs a dedicated pre-action shape (separate `forbidden_capabilities` list or a new rule type) when v0.2 introduces HITL flow. Shipping a structural no-op preserves the evaluation call site for forward compatibility.

**Impact:** Gate is two-state: `"permitted"` or `"forbidden"`. Constitutional rules still load and `forbids()` is still called — the call site is correct, the v0.1 implementation just returns False. Tests explicitly cover this no-op behavior.

## DEV-010: requires_review decision deferred to v0.2

**Spec:** `v01_phase6_design.md §5.2` describes a `requires_review` decision and `r.requires_review(proposed_action)` predicate on `LeewayRule`.

**Deviation:** `requires_review` path is not implemented in v0.1. `LeewayRule` has no `requires_review_capabilities` field or `requires_review()` method. `GateDecision.review_required_by` retains its `field(default_factory=list)` default so the v0.2 schema extension is non-breaking.

**Rationale:** No HITL (human-in-the-loop) flow consumer exists yet — the action protocol that handles a `requires_review` gate decision belongs to Phase 9+. Producing the decision state with no consumer is a no-op that adds test surface for an unreachable code path. A clean two-state gate now is better than a partial three-state gate.
