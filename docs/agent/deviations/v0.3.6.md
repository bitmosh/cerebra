## DEV-026 — cascade_depth field added to ClutchDecision and ClutchDecisionMade event payload

**What changed:** `ClutchDecision` gained `cascade_depth: int = 0` (default 0, backward compat). `ClutchDecisionMade` event payload now includes `cascade_depth` and `escalate_to_catalyst` fields. Two construction sites in `cycle_runtime.py` updated.

**Why:** Observability. Without cascade_depth, there's no way to know whether a cycle accepted quickly (first rule) or deep in the cascade (fifth rule). This matters for catalyst escalation logic in Step 2 and for debugging rule ordering.

**Impact on fossic vocabulary:** `ClutchDecisionMade` event shape extended. New fields are additive (existing consumers that don't read them are unaffected). `docs/aseptic/cross-pollination/pass-9.1.md` notifies fossic of the vocabulary extension.

**Scope:** Non-breaking. Default value (0) means any code reading old event payloads without the field will get a missing-key error if it assumes the field exists, but no current consumer reads `cascade_depth` from stored events.

## DEV-027 — composite_floor field added to CycleConfig (non-breaking, default 0.3)

**What changed:** `CycleConfig` dataclass gained `composite_floor: float = 0.3`. `_parse_config` reads it from YAML via `data.get("composite_floor", 0.3)`. `CycleRuntime` reads `self.config.composite_floor` for per-step `consecutive_steps_below_floor` tracking.

**Why:** The `consecutive_steps_below_floor` predicate needs a reference floor per cycle. Making it a cycle-level config field allows per-cycle tuning (e.g., critical cycles use 0.5; exploratory cycles use 0.2). The stop condition `composite_floor_consecutive` continues to use its own `threshold` parameter — two independent concepts with coincidentally similar semantics.

**Impact:** All existing cycle configs that lack `composite_floor` continue to parse and run without modification. `simple.planning.v0.yaml` was not modified. Default value (0.3) matches the value previously hardcoded in stop condition parameters.

## DEV-032 — Bandit implementation: RNG is stored but not yet used in v0.1

**What:** `Bandit.__init__` accepts `rng: random.Random | None = None` and stores it as `self._rng`. Per the §11 spec, the RNG is used only for tiebreaking, which is "not implemented in v0.1 — first-wins for ties is acceptable." The `select` method does not call `self._rng` in any code path.

**Why recorded:** The injected-RNG pattern is correct and necessary for future weighted-random selection (Step 3's Catalyst). The field exists at the right level. The current select implementation is deterministic without the RNG because: (1) forced_exploration always picks the first unseen arm by declaration order, and (2) UCB selection uses strict `>` comparison so ties resolve by first-encounter order without randomness.

The determinism test (`test_same_seed_same_operations_produce_identical_state`) still passes because the full operation sequence (update → select → update → select) produces identical state through pure order-preservation, not through seeded randomness.

**Impact on consumers:** Step 3 (CatalystEngine) will use `get_stats()` to read arm statistics rather than calling `select()` directly. The RNG will become relevant if/when the Catalyst or a future consumer needs weighted-random selection among UCB-equivalent arms.

## DEV-038 — Bandit arm_stats inheritance: snapshot-at-spawn via fallback load

**What changed:** `CatalystEngine.__init__` gained `parent_session_id: str | None = None`.
`_load_bandit_state()` tries own session_id first; if no rows found, falls back to
`parent_session_id`. This implements the "snapshot-at-spawn" pattern: child loads parent's
stats at construction time, then accumulates to its own rows independently.

**Why this matters:** Without inheritance, each child session starts with a fresh bandit
(all arms at count=0, forced-exploration for all 5 arms). With inheritance, the child
immediately benefits from the parent's learning history. For v0.1 where planning.adaptive.v0
has only 5 arms and max 12 steps per cycle, the difference is significant for strategy
selection quality.

**Scope boundary:** `CycleRuntime.__init__` now passes `session.parent_session_id` to
`CatalystEngine`. This is the only wiring point — no change needed elsewhere.

## DEV-039 — Synchronous child spawn: signal handler overwrite

**What changed:** `CycleRuntime.__init__` installs SIGINT/SIGTERM handlers. When a child
`CycleRuntime` is constructed in `_try_reinject()`, it overwrites the parent's signal handlers.
After child completes, the original parent's handlers are not restored.

**Why this is acceptable in v0.1:** The parent has already terminated by the time the child
is constructed (the signal handlers are no longer needed for the parent's execution). The child
is the active execution context at that point. If the user interrupts during child execution,
the child handles it.

**v0.2 note:** If child session spawning becomes async (queued, not inline), the signal
handler pattern needs revisiting. For now, sequential: parent exits → child installs → child
exits → nothing active.

## DEV-040 — ReinjectionTriggered payload: child depth, not parent depth

**What changed:** `ReinjectionTriggered.recursion_depth` contains the CHILD's depth
(`child_session.recursion_depth = parent.recursion_depth + 1`), NOT the parent's depth.

**Per S4-D7 canonical convention:** The depth in the payload is the spawned child's depth.
A `ReinjectionTriggered` at the top of the chain has `recursion_depth: 1` (child is at depth 1,
parent was at depth 0). A second re-injection has `recursion_depth: 2`. This mirrors the
`SessionOpened.recursion_depth` convention.

**Why recorded:** The convention (child depth vs parent depth) is a footgun for anyone building
Lattica visualizations of re-injection chains. Documented explicitly here and in pass-9.4.md.

## DEV-035 — Catalyst reward not recorded on LLM failure steps

**What changed:** When an LLM call fails (both retries fail), CycleRuntime resets `pending_catalyst_selection = None` and `pending_strategy_hint = ""` before entering the Clutch path. This means the bandit does NOT receive a reward update for that step.

**Why:** LLM failures produce composite=0.0 by convention, but that 0.0 doesn't reflect actual quality of the catalyst arm's strategy — it reflects infrastructure failure. Training the bandit on these synthetic 0.0 rewards would unfairly penalize whatever arm was selected. The bandit learns only from steps where the LLM actually ran and the evaluator scored the output.

**Impact:** Arms selected before an LLM failure remain "sampled" by the bandit's count only if `record_reward` was already called for them (it wasn't). So after an LLM failure, the arm that was in `pending_catalyst_selection` before the failure will be re-explored next time as an unsampled arm (count=0). This is a minor behavioral quirk but is preferable to poisoned reward data.
