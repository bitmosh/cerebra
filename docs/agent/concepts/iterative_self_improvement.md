# Iterative Self-Improvement

*Concept document — drafted 2026-06-09 following the evaluative frame doc. Status: open exploration, post-v0.1 architectural direction. Not implementation scope; companion to the interpretive lattice, archetypal lenses, and evaluative frame documents.*

---

## What this concept addresses

The lattice, the lenses, and the frame describe a cognitive architecture. They specify how Cerebra interprets content, how it commits to interpretations, and how it checks those commitments for coherence. None of the three documents addresses what to do with the resulting system over time — how the model that runs the architecture gets better at running it, and how the architecture itself evolves as the model's capabilities grow.

That is what this document addresses. The architecture produces a kind of training signal that ordinary supervised fine-tuning cannot extract from labeled examples. Multi-commit footprints across the lattice carry information about which interpretations served which queries. Lens transformation artifacts carry information about which frame-shifts produced productive reinterpretations. Frame deviation-compensation chains carry information about how the system rebalanced itself when individual aspects drifted. Silence operator activations carry information about when forced commitment would have been worse than abstention. Together, this is not a labeled dataset. It is structured cognitive experience, and it is exactly the kind of signal that single-task SFT pipelines do not know how to use.

The methodology described here is how to extract that signal, turn it into training corpora, and progressively improve the model that runs the architecture. The pattern is iterative: each cycle of system-operation produces data, the data becomes training material, the model improves, the improved model produces richer data, and the architecture's harness can be extended into territory the previous model could not handle. The cycle is sometimes called bootstrapping in the ML literature, sometimes called harness internalization, and is closely related to but distinct from what Anthropic and other labs mean when they discuss distillation. The terminology matters and is addressed below.

## The pattern, named carefully

The pattern, structurally, is a feedback loop where a model operating under cognitive scaffolding produces data that is then used to train that same model (or its successor) to do more without the scaffolding. The model learns from its own scaffolded behavior; capabilities migrate from harness code into model weights; the harness is freed to take on harder work that the new model still cannot do alone. This repeats.

The closest established term for this in the literature is *iterative self-distillation* or *bootstrapped fine-tuning*. Both terms are accurate at a technical level. Both, however, share a word — distillation — that is now flagged in production AI safety contexts. Anthropic's Fable 5 release explicitly includes a classifier that detects distillation attempts and routes them to fallback models. The classifier's framing is broad: it triggers on patterns that look like extracting frontier-model capabilities to train competing models. Cerebra's methodology is categorically different from that worried use case, but the terminology overlaps in ways that could produce false positives if the wrong frame is used.

The recommended internal terminology is *harness-internalization spiral* or, for the data-preparation portion specifically, *corpus enrichment via strong-model amendment*. Both names describe what is actually happening: the model is operating under a cognitive harness, the harness's outputs are being prepared as training data with assistance from stronger models, and the resulting training shifts harness-mediated capabilities into the base model. None of this is extracting frontier capabilities to train a Claude competitor. The end artifact is a domain-specific cognitive classifier running on a Granite-class base, not a general-purpose model. The naming should reflect that.

This is not pedantry. Names shape how work gets received, by classifiers and by humans. The work is legitimate; the framing should make it obvious.

## What gets internalized, iteration by iteration

The architecture produces several distinct kinds of structured data, and each kind becomes a different stream of training signal.

The lattice produces *interpretation-with-usage records*. A chunk multi-committed to three categorical paths gets retrieved differently for different queries; over time, the system learns which interpretation served which query. This is not "the correct label for this chunk" — it is "for chunks of this shape, the DESIGN interpretation was useful for queries of this kind, while the PRINCIPLE interpretation was useful for queries of that kind." Training on this kind of data produces a model that classifies content with implicit awareness of how the classification will be used downstream. That is qualitatively different from training on flat labels.

The lens system produces *frame-shift artifacts*. When the Builder lens interprets a chunk as DESIGN and the Critic lens reinterprets it as PRINCIPLE, the *motion* between those views is a recorded event. Over time, the system accumulates a record of which lens transitions produce productive reinterpretations for which kinds of content. Training on this kind of data produces a model that can do frame-shifting natively — recognizing when a particular kind of content warrants reinterpretation from a different posture, without needing the explicit lens harness to drive it. This is a capability that does not exist in single-classifier fine-tuned models.

The frame produces *deviation-compensation chains*. When a commit decision shows overconfidence relative to available evidence, the frame triggers Right-Context to gather more information, and the eventual outcome (well-calibrated commit, abstention, or rejection) becomes a recorded chain. Training on this kind of data produces a model with internalized metacognitive sensitivity — recognizing on its own when its confidence pattern has drifted from its evidence pattern and needing to adjust. Most models cannot do this even with explicit prompting; training on chains the frame produced is one of the few ways to actually instill it.

The silence operator produces *abstention rationales*. When silence wins on a chunk, the system records *why* — which lens activations failed to produce confident commitments, which evaluative aspects flagged forced-fit, what kind of content the chunk turned out to be. Training on this kind of data produces a model that knows its own limits — that abstains when abstention is warranted, rather than producing forced classifications because that is what its training distribution rewards.

Together, these four data streams produce a meta-cognitive training corpus that single-task pipelines cannot generate. The methodology's job is to extract them efficiently, prepare them for training, and feed them back into the model in a way that the model can actually learn from.

## The mechanism: weak teacher with strong corrector

The bulk of training data extraction happens through the architecture's normal operation. The model runs with the harness in place; the harness produces structured data as a side effect of operating; that data accumulates in the substrate. Most of the data does not need any external annotation — it is already structured, already labeled by use, already inspectable.

What requires external work is *correcting* the data where the harness made mistakes. Above-threshold-but-wrong commits need amendment that names the attractor the model fell into. Below-threshold uncertainties need explanation of the moves that would have resolved the ambiguity. Frame compensations that did not actually resolve the deviation need diagnostic notes about why the compensation failed. These corrections produce the highest-quality training signal in the corpus, and they require something the harness cannot do for itself: a stronger reasoning agent that can see the harness's mistakes and articulate the correction.

The proposed mechanism is a two-tier teacher arrangement. The weak tier — Granite 4.1 3B Instruct, or whatever instruct model the deployment uses — handles bulk labeling of the routine corpus. The weak tier runs frequently, at low cost, on Cerebra's existing hardware. Its outputs are accurate enough most of the time to produce a usable training corpus directly, and where it is inaccurate, the inaccuracies tend to be in predictable directions that other parts of the system can detect and flag.

The strong tier — Opus 4.7 at high effort, or a comparable frontier model — handles correction of a smaller subset. The subset is drawn from cases the weak tier flagged as low-confidence, cases the frame's deviation-compensation chains identified as problematic, and cases sampled randomly to provide calibration baselines. The strong tier produces amendments that include the structural reasoning the weak tier could not articulate. These amendments do not replace the weak tier's outputs; they augment them with the additional signal that makes the corpus train better than the weak tier alone could produce.

This is not distillation in the worried sense. The strong tier is not being copied. Its outputs are not being used to train a model that would replicate its general capabilities. The strong tier is being used as a *sophisticated annotation tool* on a corpus that is bounded to a narrow task domain (SKU classification, lens evaluation, frame coherence checking). The end artifact is a fine-tuned classifier for that specific domain, not a frontier model surrogate. The relationship to standard distillation work is structural rather than substantive: the same machinery is being used for categorically different purposes.

## The hard-example mining loop

The lattice's two failure pools — above-threshold-wrong and below-threshold-uncertain — receive different corrective treatments. Both treatments require the strong tier; both produce training signal that is more concentrated than the routine corpus; both pay for their cost by improving the model's performance on the cases where it currently fails.

For above-threshold-wrong cases, the corrective pass produces what could be called *error-attractor surgery*. The strong tier examines the wrong commit, identifies what surface feature of the chunk likely drew the model toward the wrong category, and writes an amendment that names the attractor explicitly. The training example then has three parts: the chunk, the model's wrong commit with its confidence, and the amendment naming the attractor with the corrected commit and explanation of why the correct interpretation should have been visible. This is qualitatively richer than "here is the right label." It teaches the model not just the right answer but the structural reason its wrong answer was tempting.

For below-threshold-uncertain cases, the corrective pass produces *resolution-strategy training*. The strong tier examines the chunk and articulates the moves the system could have made to resolve the ambiguity: which surrounding chunks would have provided disambiguating context, which lens activations the metacognitive selector should have engaged, which evaluative aspects should have triggered to focus the system's attention. The training example has the same three-part structure: the chunk, the model's abstention or weak commit, and the strategy that would have produced a confident commit. This teaches the model not just what to do, but how to *get to* what to do when the initial path is unclear.

These corrected hard examples should be weighted higher than routine examples during training. Their information density is higher and they target the cases where the model is currently failing. The standard practice in hard-example mining is to weight by 2× to 5× depending on how concentrated the failure cases are; the same heuristic applies here. The exact weighting is a calibration concern that emerges from training experiments rather than being knowable in advance.

A specific variant worth naming is *synthetic-exemplar steering*. For each category, the strong tier produces a small set of picture-perfect synthetic examples — chunks crafted to embody the category clearly, with no ambiguity. These synthetic exemplars are not added to the corpus directly. Instead, they anchor the amendments: when the strong tier corrects a below-threshold-uncertain case, the correction is shaped to move the model toward the exemplar's style of clarity. The exemplars act as *attractors in the corrective direction*, the way the wrong surface features in error cases are attractors in the wrong direction. Done with care, this preserves diversity (multiple exemplars per category, drawn from different stylistic regions) while still providing clear teaching signal.

The risk to mitigate is style homogenization. If the exemplars become a small set of identical-feeling chunks, the amendments steer the corpus toward a narrow stylistic region and the model loses diversity in its outputs. Mitigation: maintain three to five exemplars per category, drawn from different stylistic regions, and use the *set* as the attractor rather than any single exemplar. The model should be steered toward a *region* of clarity rather than a *point*.

## Drift amplification, the central failure mode

The pattern of training a model on outputs from the previous iteration of that same model has a well-known failure mode in the literature. Each iteration's training corpus reflects the previous iteration's strengths and weaknesses; any systematic error in iteration N gets reinforced in iteration N+1; without external anchoring, the system drifts away from ground truth over time. The drift can be subtle — the model gets more confident, more fluent, more internally consistent — while becoming worse at the actual task it is trying to do. This is sometimes called model collapse or training collapse, depending on how acute the drift becomes.

The mitigation has two components. The first is *held-out evaluation* against data the system has never seen and that was not produced by any version of the model. Each iteration is evaluated on the same held-out set; if the iteration's performance on the held-out set drops, the iteration is treated as a regression and discarded, even if it improved on the training distribution. This catches the most obvious drift patterns.

The second component is more demanding: *arbitrary-content evaluation* against data that lies outside the training distribution entirely. The held-out set described above is still drawn from the same kind of content the system was trained on; arbitrary content is deliberately drawn from kinds the system has never been exposed to. If iteration N performs well on familiar held-out data but poorly on arbitrary content, the system has begun to overfit its own distribution. The mitigation is to broaden the training corpus before continuing, not to push further along the current trajectory.

The arbitrary-content evaluation set is constructed across three tiers. The in-distribution tier contains content from the system's existing corpus, classified by humans, used as the primary benchmark. The adjacent-domain tier contains content from related but unseen sources — other people's planning documents, open-source architecture documents, technical writing in adjacent fields. The out-of-distribution tier contains content that is genuinely unlike the training set — news articles, fiction, scientific papers from unrelated disciplines, philosophical writing. Each tier tests a different generalization property: in-distribution tests routine performance, adjacent-domain tests within-style generalization, out-of-distribution tests the limits of the system's reasoning when surface features differ.

A particularly informative class of test material is *paradox-shaped content*. These are constructions where the apparent question has no clean answer because the question's structure rules out the available answers — koan-shaped material, in its functional sense, scrubbed of any religious framing. A system that performs well on routine classification but produces forced classifications on paradox-shaped content is a system that has not actually internalized abstention; it has only learned to abstain on patterns that look like abstention cases in its training distribution. A system that recognizes paradox-shaped content and produces silence-operator activations with appropriate rationales is a system whose abstention capability has generalized. These tests are not graded for "right answer" — there often is none. They are graded for *response category*: did the system produce a forced fit, a meaningful refusal, or a productive reframe of the question itself? The third category is the highest mark and the hardest to achieve. A system that can reframe paradox is doing something most ML systems demonstrably cannot.

## The realistic horizon

The spiral has finite useful range. After two or three iterations, the gains from each subsequent cycle diminish substantially. The easy patterns get internalized first; what remains is increasingly hard cognitive work that the model may never fully absorb regardless of how many iterations are run. This is not a failure of the methodology; it is a property of how learning works in systems with bounded capacity.

The right way to think about the horizon is in two phases. The first phase, lasting two or three iterations, produces large capability gains: the model becomes substantially better at running the architecture, requires substantially less harness intervention, and demonstrates capabilities that the initial model could not produce. This is where the methodology pays its largest returns and where most of the effort should be concentrated.

The second phase, after the initial phase plateaus, is qualitatively different. Further iterations produce small gains, but the small gains require disproportionate effort. The harness work that remains is work that the model genuinely cannot internalize — work that depends on the harness's structural commitments rather than on patterns the model could learn to reproduce. This is where koan-shaped content becomes diagnostic: a model that has plateaued on the methodology may still be improving on routine classification tasks, but its handling of paradox-shaped material does not improve any further. That is the signal that the architecture has hit the limit of what training can move into weights, and the remaining work is permanent harness rather than transitional scaffolding.

Both phases produce value. The first produces a substantially stronger base model. The second confirms which parts of the architecture are durable contributions rather than training-time scaffolding. The methodology is not failed when the second phase begins; it has transitioned into its sustainable-running mode, where the harness is performing the work it will always perform and the model is supporting that work with the capabilities it has gained.

## Train-and-retire cycles

A subtle methodology decision concerns whether training corpora are cumulative or rolling. Cumulative training keeps every example ever produced in the training set, on the theory that more data is always better. Rolling training retires older corpora periodically, on the theory that they reflect older versions of the system and may not serve newer ones.

The right approach for Cerebra is rolling. Cumulative training corpora develop subtle pollution over time: older examples reflect classifications the system would now make differently, lens activations the system would now route to different lenses, evaluative deviations the system has since learned to avoid. Training on the pollution teaches the model to reproduce behavior that is now suboptimal. The pollution is invisible — the corpus looks fine, the training metrics look fine — but the model's capabilities erode in ways that take time to notice and longer to undo.

The proposed cycle is corpus retirement after each successful iteration. The corpus that produced iteration N is archived (preserved for audit and analysis) but does not contribute to training iteration N+1. Iteration N+1 trains on a fresh corpus produced under iteration N's harness, with whatever corrections and amendments the strong tier provided. The carry-over from one iteration to the next is the *model*, not the *corpus*. Each iteration starts with a fresh corpus and ends with a stronger model.

This produces a clean training trajectory but requires discipline. The temptation to retain older corpora is real, because they represent effort that has been put in and seem like assets to preserve. The argument for retirement is that the *value* of the older corpora is preserved in the model that was trained on them; that value persists across the spiral regardless of whether the corpora themselves are kept active. Archiving rather than deleting handles the desire to preserve audit material without paying the pollution cost.

## Connections to related concepts

The interpretive lattice provides the data substrate the methodology trains on. The multi-commit footprints, the usage labels, the interpretation-to-query relationships — all of this is what the methodology extracts as training signal. Without the lattice, the methodology has only single-label classifications to work with, which is the kind of signal that ordinary SFT already handles well. The lattice is what makes the methodology produce meta-cognitive training signal rather than task-specific training signal.

The archetypal lenses provide the frame-shift training stream. The transformation artifacts the lens system produces are inputs the methodology can use to train a model that does frame-shifting natively, without needing the lens harness to drive it. Each iteration of the spiral internalizes some of the lens-mediated frame-shifting into the model itself; later iterations of the harness can introduce new lenses that the previous model could not handle and could not be trained on.

The evaluative frame provides the meta-cognitive training stream. The deviation-compensation chains the frame produces are inputs the methodology can use to train a model that recognizes its own miscalibrations and corrects them without external intervention. This is the single most valuable capability the spiral can produce, because it is the capability that lets the model run with progressively less harness support over time.

The silence operator provides the abstention training stream. The chunks routed to the uncertain pool and the rationales the silence operator produces are inputs the methodology can use to train a model that abstains when abstention is warranted. This is the capability that makes the model trustworthy on adversarial or paradox-shaped content, and it cannot be produced by training on standard supervised corpora because those corpora reward output rather than appropriate non-output.

Together, the four architectural documents and this methodology document describe a complete loop. The architecture produces specific kinds of structured data. The methodology converts that data into training corpora. The training improves the model. The improved model runs the architecture better. The architecture's harness can be extended into new territory the previous model could not handle. The loop repeats until the gains diminish, at which point the architecture's permanent contribution becomes visible.

## Costs and mitigations

The strong-model API costs are the most concrete. Each iteration requires the strong tier to produce amendments on a non-trivial fraction of the corpus. At Opus 4.7 high-effort pricing, the cost per iteration for a corpus of several thousand records is substantial — not prohibitive for a single developer running periodic iterations, but enough that the iteration cadence has to be deliberate rather than continuous. Mitigation: triage the strong-tier corrective work aggressively. Only the failure cases (above-threshold-wrong, below-threshold-uncertain) need strong-tier amendments; the routine corpus can pass through with weak-tier labeling alone. This reduces the strong-tier load to perhaps 10–20% of the total corpus per iteration.

Time per iteration is the next cost. End-to-end, each iteration spans corpus extraction, strong-tier amendment, training preparation, fine-tune execution, evaluation, and the decision about whether to keep or discard the iteration's output. Realistic timeline per iteration is one to two weeks of focused work, assuming the surrounding infrastructure (training pipelines, evaluation harnesses, corpus management tools) is in place. The initial iteration takes longer because the infrastructure is still being built; later iterations are faster.

Methodology drift across iterations is a more subtle cost. The methodology itself has parameters — the confidence threshold, the failure-pool weighting, the strong-tier sampling rates, the synthetic-exemplar set composition — and these parameters can drift across iterations in ways that change what the methodology is actually doing. Mitigation: version the methodology explicitly. Each iteration runs under a named methodology version; changes to the methodology produce a new version; the version history is preserved so iterations can be compared meaningfully. This is the same discipline Cerebra applies to its forward-only migrations — methodology changes are forward-only as well, and reproducibility depends on the version being identifiable.

The risk that the strong tier itself becomes biased toward a particular interpretation style is real and worth flagging. If the strong tier consistently amends in a particular direction — toward more structured outputs, toward more conservative classifications, toward a particular vocabulary — the corpus carries that bias and trains the model toward it. Mitigation: vary the strong tier across iterations where possible (Opus 4.7 in one iteration, Opus 4.8 in another, perhaps a different family in a third). The variance across strong tiers acts as its own regularization against any single tier's idiosyncrasies.

## Open questions

The right cadence between iterations is the first. Too frequent and the model never stabilizes; too infrequent and the spiral takes years to produce useful results. The proposal is monthly to quarterly cadence depending on corpus accumulation rate and on whether evaluation indicates iteration N's outputs have stabilized. This needs empirical refinement.

The right weighting between routine corpus and hard-example corpus is the second. Hard examples are more information-dense but also more idiosyncratic; over-weighting them risks training the model to over-fit on the specific failure cases that were corrected. The standard ML practice of 2× to 5× weighting is a starting point, but the actual ratio for Cerebra's specific corpus structure has to come from experimentation.

The role of human review in the spiral is the third. The methodology as described relies entirely on machine-generated training data with strong-model amendment. Human review could be added at several points — reviewing strong-tier amendments before they enter the corpus, reviewing iteration outputs before they replace the previous model, reviewing arbitrary-content evaluation results. The tradeoff is between methodology speed (less human review is faster) and methodology trustworthiness (more human review catches subtle drift earlier). For a solo-developer context, light human review at decision points seems right.

The relationship to community-provided training data is the fourth. As Cerebra is eventually shared with other people, those people's vaults produce their own multi-commit footprints, lens activations, and frame chains. Whether and how to aggregate community-produced training data into shared model improvements is a real question that has not been engaged here. The decision is partly technical and partly about privacy and ownership commitments that have not been specified yet.

The convergence point of the spiral is the fifth. At what number of iterations does the system stop benefiting from further iteration? The estimate of two to three is based on general bootstrapping literature, but Cerebra-specific data will produce a more accurate answer. The question matters because it shapes how much sustained investment the methodology can absorb before transitioning to its sustainable-running mode.

## What this document is and isn't

This is a concept document, not an implementation specification. It articulates the methodology that ties Cerebra's cognitive architecture to its training trajectory. The implementation depends on choices that will be made later — exact training pipeline mechanics, evaluation infrastructure, strong-tier integration, version control for methodology parameters — and on capabilities that do not yet exist (the lattice, the lenses, and the frame are all currently concept-only).

Implementation is post-v0.1 and follows the lattice, lens, and frame implementations. The methodology can begin in a limited form earlier — corpus extraction from the existing single-classifier system, amendment pass with the strong tier, fine-tuning with the resulting corpus — and several of these limited-form steps are already proposed in the deferred v0.2 LoRA track. The full methodology emerges as the architectural components become available.

Expected scope when full implementation begins:

- Corpus extraction tooling that pulls structured data from the lattice, lens activations, and frame chains
- Strong-tier integration with rate limiting, cost tracking, and amendment quality monitoring
- Synthetic-exemplar generation and maintenance for each lens and category
- Training pipeline that handles the multi-stream data appropriately (categorical, transformation, deviation-compensation, abstention)
- Evaluation infrastructure for three-tier arbitrary-content testing including paradox-shaped fixtures
- Methodology version control and iteration management

Estimated implementation scope: substantial, distributed across multiple Cerebra phases as the architectural components come online. The methodology document exists so the eventual implementation work has a coherent target rather than developing ad hoc as each architectural piece lands.
