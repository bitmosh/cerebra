I — The Arc

What's been built, in order of how it became necessary.

Cerebra started as source-memory infrastructure and accumulated layers as each layer's absence became the next constraint. The work moves in phases — design doc, then implementation, then close-out — with version markers at every meaningful threshold. The architecture you're reading wasn't planned all at once. It surfaced.

v0.0.1 — v0.1.0 Phases 1–2 SHIPPED

Foundations

Source ingestion, chunk-and-record architecture, and a working SKU classifier. The substrate the rest of the system would sit on.

Phase 1 established the source memory model: documents enter as sources, get chunked into chunks, each chunk becomes one or more memory records. The forward-only migration discipline started here and has held through every subsequent phase.

Phase 2 shipped the production substrate at v0.1.0: Granite 4.1 3B Instruct as the SKU classifier, running locally via Ollama. 65% partial-credit accuracy on the v0.1.0 calibration set. The methodology question that surfaced — multi-model comparison to surface contested classifications — became the foundation for everything that came after.

The benched LoRA arc lived between Phase 2 and Phase 3. Phase 1 of training hit 10% with 83% parse failures. Phase 2 of training fixed the methodology (EOS + completion-only loss) and reached 46.7% with 0 parse failures. Phase 3 of training surfaced the real problem: corpus imbalance. The work was strategically benched pending corpus growth or a specific downstream need — a deliberate scope cut, not a failure.

Banked

Forward-only migrations as an inviolable discipline. Edit-in-place breaks reproducibility; subsequent migrations with idempotent guards is the only acceptable path.
The two-tier teacher pattern (weak teacher + strong corrector) emerged as the right methodology for corpus enrichment, with naming separated from "distillation" to avoid classifier overlap with worried use cases.
v0.1.0 — v0.1.5 Phase 3 SHIPPED

Storage and the Index Layer

The substrate gained the indexes it needed to be queryable: vector embeddings, lexical FTS, a polymorphic graph schema, and event-driven freshness tracking.

The phase shipped five new tables in Migration006 — embeddings, pending_embeddings, index_state, graph_nodes, graph_edges — plus Migration007 for seeding fixes after a real-world wrinkle (Migration006 was already applied to the dev vault, so its seed logic moved to a new migration rather than editing the original).

The graph schema is polymorphic from day one: 20 structural edge types plus 16 SKU semantic edge types are schema-accepted in Phase 3, even though Phase 3 only writes two of them. The decision to leave the doors open up front costs nothing in migration weight and avoids expensive schema changes later when more edge types get used.

The embedding model is mxbai-embed-large-v1 at 1024-dim, with Matryoshka representation learning preserved. 745 records embedded in 9.6 seconds on RTX 4070 SUPER, 11.9ms average latency. Forward-compat columns reserved for future training work (embedding_alt).

Verification ran 8 vector queries against the corpus: 0.696 mean top-1 on domain queries, 0.485 on adversarial. A 0.21 discrimination gap. Clean separation between "system knows this" and "system doesn't know this."

interlude Cognitive Extension CONCEPT

Five concept documents.

Between Phase 3's close and Phase 4's design, a four-hour conversation produced a coherent post-v0.1 architectural direction. The concept docs are durable artifacts, not implementation specs.

The series describes a layered cognitive architecture extension: a substrate that preserves interpretive ambiguity, a perspectival layer that generates multi-commit signal, a coherence layer that keeps commits honest, and a training methodology that converts the resulting structured data into model weights. Forward-compat with current Cerebra; nothing in the concept docs blocks shipped work.

Interpretive Lattice

interpretive_lattice.md · substrate

Confidence-gated multi-commit. When the classifier produces a distribution where multiple categories clear 65%, the chunk lands at every qualifying position. Interpretive ambiguity preserved rather than collapsed.

Archetypal Lenses

archetypal_lenses.md · generation

Twenty-two cognitive postures evaluated in parallel, with full N×N transformation operators between them. Silence operator as a first-class lens. Structural pattern from Major Arcana, content removed.

Evaluative Frame

evaluative_frame.md · coherence

Eight mutually-reinforcing aspects of cognitive functioning, with clutch-driven event-based rebalancing. Constitutional self-critique pattern applied to interpretive coherence rather than harm avoidance.

Self-Improvement

iterative_self_improvement.md · method

Harness-internalization spiral: model runs with cognitive harness, structured data captured, strong-corrector amendments applied, model fine-tuned, harness extends into new territory. Drift mitigated by held-out paradox-shaped fixtures.

A fifth document, cognitive_extension_overview.md, ties the four together. Total ~15,600 words. Implementation is post-v0.1, distributed across future phases. The docs exist so the architectural commitments can be made deliberately rather than discovered during implementation.

v0.1.5 — v0.2.0 Phase 4 SHIPPED

Retrieval Orchestration

The substrate started doing something visible. A query enters; six steps run; ranked results emerge with full provenance. Abstention semantics catch the cases where the right answer is silence.

The query planner classifies incoming queries into D1 categories via a heuristic keyword vocabulary (loadable TOML, no LLM cost). Three modes — hybrid, lexical_only, vector_only — selected from query shape.

The six-step traversal walks the SKU substrate: query construction, exact SKU match, partial D1 match, sibling traversal (explicit no-op for v0.1 with documented swap-in interface), vector fallback (200-cap), trace annotation. Every result carries its retrieval path — a string describing how it was found. partial_sku:D1=0xe + vector_fallback is real output.

Salience composition runs five components with calibrated weights: semantic 0.40, lexical 0.25, sku_match 0.15, recency 0.10, lifecycle 0.10. The lexical normalization formula was direction-pinned by unit test after a near-miss where the formula was inverted in the design doc. Best BM25 maps to 1.0; worst maps to 0.0. The test catches any future re-inversion.

Abstention is the substrate-level silence operator. When the maximum salience score falls below the configurable floor (default 0.35), no commitment happens. A structured abstention packet is returned containing best_score_seen, an abstention_rationale, and an empty selected_memory list. Downstream agents get an honest answer to "what should I tell you when retrieval found nothing useful?"

ContextPackets carry full provenance: retrieval_trace_id, origin_event_ids, per-candidate score components, vault-relative source paths, packet_version for forward compat. The packet is regenerable from the trace, not stored. The trace is queryable for any retrieval that ever happened.

v0.2.0 — v0.2.2 → Phase 5 IN PROGRESS · 4/8

Working Memory and Truth Tower

The substrate gains genuine state across time. Working memory persists between command invocations. A two-tier truth tower captures what the system has actually verified.

Phase 5 is the threshold where Cerebra becomes "an actively thinking system" rather than "a substrate that supports thinking." Through Step 4, working memory operates as ten named slots (goal, constraint, context, hypothesis, evidence, contradiction, recent_output, question, procedure, interrupt) with capacity caps and a four-tier eviction policy: pinned items never evicted, tower-cited items eviction-resistant, lowest effective salience evicted first, oldest as tiebreaker.

Sessions are persistent — one active per vault, generated on first write-path command, reusable across subsequent commands. The lockfile mechanism from Phase 3's design doc (which got carried as debt through Phase 4) finally landed in Step 2 with proper PID-based stale detection.

The truth tower (Step 5+) will hold T1 evidence (auto-populated from cerebra context) and T2 insights (manual promotion with T1 citation requirement). Staleness propagation when T1 evicts. Chronological render embeddable in ContextPacket via the optional truth_tower field — no packet version bump needed.

Forward-compat columns reserved on working_memory_items at Migration009 time: interpretive_lens and frame_metadata_json, both nullable in Phase 5. The cognitive extension can be implemented later without further schema changes to this table.

II — What's Ahead

The work remaining, in the order it earns its place.

The roadmap beyond Phase 5 is mostly composing what's already there into a working cognitive runtime. Each phase has a single thesis and a clear set of architectural commitments. The pace from Phase 3 forward suggests these phases land in roughly 1–3 weeks each.

06·

Cycle Runtime

the thinking loop

Working memory + truth tower + retrieval composed into reasoning cycles. A cycle starts with a goal, retrieves into working memory, derives toward the tower, evaluates, consolidates. This is where Cerebra goes from "you can ask it" to "it can ask itself."

DELIVERS: cerebra serve daemon mode; cycle_id as runtime context; the actually thinking demo.
07·

Prediction and Evaluation

did we get it right

The runtime records what it predicted and whether the prediction held. Each cycle's outputs become training data for whether the cycle's reasoning was sound. This is the layer that lets the system improve from its own operation.

DELIVERS: PredictionMade and PredictionEvaluated events as structured training signal; calibration of confidence vs accuracy at the cycle level.
08·

Consolidation

long-running memory hygiene

Memory records age. Tower items get evicted or revisited. Some interpretations get retired in favor of better ones. Consolidation is the policy layer for "what happens to memory over time" — archive, tombstone, re-classify, summarize.

DELIVERS: retired-but-not-deleted commits as training signal; tower stale-management; the resolution mechanism for multi-commit ambiguity.
09·

Catalyst

strategic decisions as first-class data

Not every chunk is the same. Some decisions are load-bearing — they shape what comes after. Catalyst makes strategic decisions first-class memory entities with their own node type, lineage, and discoverability.

DELIVERS: Decision node type populated; "what did we decide and why" becomes a queryable structure rather than buried prose.
10·

Truth Tower T3–T5

insights, hypotheses, goals

The two-tier tower from Phase 5 grows to its full form. T3 is derived insight (cross-references multiple T2s). T4 is hypothesis (requires deliberate cognitive work to form). T5 is goal — set externally, never auto-promoted.

DELIVERS: full truth tower; asymmetric promotion semantics (higher tiers require deliberate work); the discipline that prevents the tower from becoming a house of cards.
III — Cross-cutting

Threads that run alongside the phase work.

Not everything fits in the phase progression. Some work is architectural cleanup, some is integration with other projects in the portfolio, some is product surface. These threads emerged from real friction and have natural homes between phases or as sidequests.

infrastructure

lattica-primitives extraction

Six primitives are vendored across Cerebra, LumaWeave, Bons.ai, Policy Scout. Extraction to a real PyPI package once 2+ consumers stabilize. Phase 6 territory or a sidequest between phases. Converts personal codebase into infrastructure others can build on.

infrastructure

eval-core extraction

Policy Scout's mature eval runner (EvalCase, runner, YAML DSL, 7 assertion types) becomes a shared package. Cerebra is the heaviest consumer (SKU accuracy, retrieval regression, LoRA quality, abstention calibration). Lives in its own repo to keep dependency directions clean.

integration

Daemon mode (cerebra serve)

Unix-domain-socket protocol with FastAPI-shaped routes. Lets Lattica's Tauri dashboard and LumaWeave's graph adapter consume Cerebra at runtime without shelling out. Phase 6's natural surface; design now so Phase 5 work doesn't accidentally close the doors.

integration

Discord ingest adapter

Discord channel history → cerebra ingest pipeline → searchable corpus. Decoupled from the discord-bot's gather_context swap (which waits for daemon mode). Probably 1–2 days of work between phases.

methodology

LoRA resume

Currently benched pending corpus growth. Phase 5+ generates new corpus shape: working memory access patterns, tower citation chains, frame deviation chains. Resume after Phase 6 ships, when usage data has accumulated enough to retrain meaningfully against arbitrary-content anchors.

methodology

Cognitive extension implementation

The five concept docs become real code starting post-Phase-5. Lattice multi-commit substrate first (extends Phase 5's working memory). Lens system next. Evaluative frame third. Iterative methodology last. Publishable as research if framed deliberately.

product

Multi-vault future

Phase 5's "one active session per vault" is a simplifying assumption. Multi-vault adds session-identity questions, permission models, sync semantics. Months of work; not v0.x; the path from personal tool to team tool to service.

product

MCP server interface

Cerebra-as-MCP-server lets other Claude or agent systems use Cerebra as their memory substrate. A real product story for the framework angle. Sits on top of daemon mode rather than replacing it.

IV — Training

The training signal each layer produces.

The architecture isn't built independent of training — it's built so that operating it produces the data needed to train better versions of itself. Each phase contributes a different kind of training signal. The methodology document spells this out at depth; what follows is the inventory.

Cerebra produces interpreted experience, not labeled examples. That's the signal nothing else generates and the reason the architecture earns its complexity.

Phase 1–2

// SKU classifications with provenance

Every chunk's classification carries its multi-model voting history. Contested classifications are queryable. Stage-2 disagreement is structured data, not noise — it's the fixture set for calibration work that hasn't started yet.

Phase 3

// embeddings + graph edges

Vector embeddings preserve semantic distance. Graph edges declare structural relationships (CONTAINS, DERIVED_FROM, plus 18 more structural and 16 SKU semantic types). The combination is the substrate for any future training that wants to learn the relationship between meaning-as-text and meaning-as-position-in-knowledge-structure.

Phase 4

// retrieval traces with usage labels

Every retrieval is fully audited. retrieval_traces + retrieval_steps + retrieval_candidates record query, plan, traversal path, candidates surfaced, scores at each component, selections, abstentions. Over time this becomes labeled-by-use training data: which interpretations served which queries, which patterns produced abstention, which queries needed which signals.

Phase 5

// working memory + tower citation chains

Access patterns over time. Which records get promoted to working memory and survive eviction. Which T1 items get cited by T2. When T1 evicts and T2 marks stale. The shape of what the system was actually holding in attention during its operation — a different signal than what it retrieved.

Phase 6+

// prediction-evaluation deltas

The runtime makes predictions; reality validates them. The delta between prediction and outcome is the training signal that lets the system improve at its own cognitive operation. This is the layer that closes the loop from "data accumulates" to "data improves the model that produces data."

Concept

// frame deviation-compensation chains

When the eight-aspect evaluative frame detects deviation in one aspect and triggers compensatory activity in others, the resulting chain — deviation, compensation, outcome — is recorded. Over time, this becomes meta-cognitive training data: how does the system recognize its own miscalibration, what compensations work, when does silence win.
